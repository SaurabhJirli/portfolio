## What it does

An event-driven pipeline that decouples services by routing domain events through Apache Kafka. Different consumers process events independently — some update databases, others trigger notifications.

## How it works

- Services produce events to Kafka topics when something meaningful happens (order placed, device connected, etc.)
- Consumers pick up events and handle them — writing to **PostgreSQL**, updating caches, or sending alerts
- Schema validation keeps event contracts consistent across producers and consumers
- Dead-letter topics catch failed messages for later inspection

## Tech used

- **Apache Kafka** for event streaming
- **Spring Boot** for producers and consumers
- **PostgreSQL** for persistent storage
- **Docker Compose** for local development

## What I learned

Thinking in events instead of direct API calls took some adjustment. The biggest lesson was around consumer idempotency — making sure processing the same event twice doesn't break anything. Also learned to appreciate Kafka's consumer group model for scaling.
